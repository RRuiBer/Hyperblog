Git (Clase 1)

Un sistema de control de versiones como Git nos ayuda a guardar el historial de cambios y crecimiento de los archivos de nuestro proyecto.

En realidad, los cambios y diferencias entre las versiones de nuestros proyectos pueden tener similitudes, algunas veces los cambios pueden ser solo una palabra o una parte específica de un archivo específico. Git está optimizado para guardar todos estos cambios de forma atómica e incremental, o sea, aplicando cambios sobre los últimos cambios, estos sobre los cambios anteriores y así hasta el inicio de nuestro proyecto.

Git Bash en Windos (Clase 3)

Al implementar Git Bash, podrás tener una terminal de Git en el sistema operativo Windows. Esto te ayudará a acceder a tus repositorios al conectarte desde GitHub.

Git Bash es una aplicación en línea que une las utilidades de Git y Bash en un sistema operativo de Windows. Mejora la experiencia de la línea de comandos y se comporta como una especie de intérprete para un sistema operativo Unix, pero adaptado para funcionar en Windows.

Además, el programa permite a los usuarios acceder a las herramientas de Git, incluyendo la versión, y realizar tareas de control de versiones a través de comandos de texto en lugar de una interfaz gráfica de usuario. También incluye otros comandos de Unix y herramientas como ssh , scp y rsync.

Pull request (Clase 28)

Es una funcionalidad de github, en la que un colaborador pide que revisen sus cambios antes de hacer merge a una rama, normalmente main.

Al hacer un pull request se genera una conversación que pueden seguir los demás usuarios del repositorio, así como autorizar o rechazar los cambios.

Crear un Fork (Clase 29)

Los forks o bifurcaciones son una característica única de GitHub en la que se crea una copia exacta del estado actual de un repositorio directamente en GitHub. Este repositorio podrá servir como otro origen y se podrá clonar (como cualquier otro repositorio).

Realizar Fork desde el repositorio de nuestro interés en GitHub.

En nuestra terminal escribimos el comando:

- git clone -direccón del repositorio que se le hizo fork-: trae los archivos del repositorio al entorno local.

Git Rebase:

Rebase reescribe la historia del repositorio; cambia la historia de donde comenzó la rama. Hacer un rebase no es una buena práctica y se recomienda usar solo en un repositorio local.

Rebase sirve también para cambiar la historia de nuestros commits en la misma rama, si tienen muchos commits con cambios pequeños estos se pueden unificar para hacer que su log se vea mas limpio.

Pasos para aplicar un rebase:

- Crear una nueva ramaa: git branch experimento

- Hacer checkout a la nueva rama: git checkout experimento

- Una vez realizados los cambios en el repositorio, guardarlos con un nuevo commit: git commit -am "Agregué información sobre el rebase".

- Si queremos juntar las ramas como si la rama experimental no hubiera existido y que lo cambios realizados queden en la rama principal. Estando en la rama 'experimento' aplicamos el rebase: git rebase main.

Nunca debes reorganizar las confirmaciones una vez que se hayan enviado a un repositorio público. La reorganización sustituiría las confirmaciones antiguas por las nuevas y parecería que esa parte del historial de tu proyecto se hubiera desvanecido de repente.

Git Stash (clase 36)

- git stash : Guarda el trabajo actual de manera temporal (Archivos modificados o eliminados).

Al ejecutar la sentencia: git stash, se regresa al estado anterior del proyecto.

- git stash list : Permite visualizar todos los stash existentes.

- git stash pop : Aplica el stash más reciente.

- git stash branch -nombre_de_rama- : Crea una rama y aplica el stash realizado mas reciente.

- git stash drop : Elimina el stash más reciente.

Git cherry-pick (clase 38)

Este comando permite llevar uno o varios commits de una rama a otra sin tener que hacer un merge completo.

En una rama distinta a la principal, después de haber creado uno o mas cambios con sus respectivos commits, podemos llevar alguno de los cambios a otra rama mediante el hash del commit.

Estando en la rama principal:

- git cherry-pick -hash del commit que queremos agregar a main-

Si posterior al cherry-pick queremos hacer un merge a la rama principal podemos generar un conflicto. Para solucionarlo, aceptamos los cambios de nuestro interés y los guardamos con un nuevo commit.

Aunque cherry-pick es útil y se puede aplicar en ramas distintas a la principal, utilizar esta función se considera una mala práctica.

Git reset y reflog (clase 39)

- git reflog: muestra el hash y head de todos los cambios, incluso de ramas que han sido borradas.

Supongamos que tenemos cambios indeseados o que se realizaron por error, además estos cambios ya tuvieron su respectivo commit.

- git reset --soft -nuero de hash-: restaura el repo al estado del hash referenciado y mantiene los cambios removidos en stagin.

- git reset --hard -nuero de hash-: restaura el repo al estado del hash referenciado y elimina totalmente los cambios que no eran deseados.

Git reset es una mala práctica. Usar solo en caso en que haya que restaurar el repositorio debido a fallas graves.

Recounstuir commits en Git con amend (clase 40)

Remendar un commit: puede modificar el commit más reciente (enmendar) en la misma rama ejecutando:

- git add -A: Este comando sirve para agregar archivos nuevos o actualizar el commit anterior y no generar commits innecesarios. También es una forma sencilla de editar o agregar comentarios al commit anterior porque abrirá la consola para editar el commit anterior.

Para hacer uso de ammend los archivos deben de estar en staging

- git commit --amend: Remendar último commit

Buscar en archivos y commits de Git con grep y log (clase 41)

- git grep -palabra-: busca las coincidencias de la palabra en el repositorio.

- git grep -n -palabra-: busca las coincidencias de la palabra en el repositorio e indica el numero de la línea donde aparece la palabra.

- git grep -c -palabra-: muestra el número de veces que aparece la palabra en el repositorio

- git grep -c "<p>": muestra el número de veces que aparece la sentencia en el repositorio (usar comillas para buscar partes de código).

- git log -S "-palabra-": muesta los commits donde apararece la palabra buscada en las comillas.

Usar grep para los archivos y log para los commits.

Comandos y recursos colaborativos en Git y GitHub (Clase 42)

- git shortlog: muesta los commits de cada usuario.

- git shortlog -sn: muestra el número de comits que realizó cada usuario.

- git shortlog -sn --all: muestra el número total de commits de cada ususario, incluso los borrados.

- git shortlog -sn --all --no-merges: muestra el número de commits de cada usuario exluyendo los commits de los merges.

- git config --global alias.-nomre del alias- "-comando que se le quiere poner alias-": asigna un alias a un comando de git.

- git blame -archivo a explorar-: para cada línea del archivo explorado muestra; muesta hash, usuario que realizó el commit, fecha y hora, número de línea y contenido de la línea.

- git blame -c -archivo a explorar-: aplica identación en el comando anterior.

- git -comando de git- --help: muesta la documentación de git con la información del comando.

- git blame -archivo a explorar- -L-numero de línea-,-numero de línea-: las lineas referenciadas del archivo a explorar muestra; muesta hash, usuario que realizó el commit, fecha y hora, número de línea y contenido de la línea.

- git blame -archivo a explorar- -L-numero de línea-,-numero de línea- -c: aplica identación en el comando anterior.

- git branch -r: muestra las ramas en el repositorio remoto.

- git branch -a: muestra tanto ramas locales como remotas.
